🌊 光核安利漂流瓶
技术框架说明 v5 "StarrySea"

一、项目简介

一个游戏安利主题的 H5 社交活动网站。用户可以投递"漂流瓶"（游戏安利卡片）、随机打捞别人的瓶子、收藏到个人安利墙、分享拉新获取奖励。

二、技术选型

层级技术
后端框架Python 3.12 + Flask（轻量 Web 框架）
数据库SQLite + SQLAlchemy ORM（对象关系映射）
前端原生 HTML / CSS / JavaScript + Jinja2 模板引擎
图片处理Pillow（上传后自动生成缩略图）
用户认证Flask-Login（会话管理，支持记住我）
总代码量约 2900 行（Python 1356 + 前端 1500+）

四、数据库设计（8 张表）

4.1 表关系总览

users（用户）
  │
  ├── 1:N → drift_bottles（漂流瓶）
  ├── 1:N → salvage_records（打捞记录）
  ├── 1:N → daily_tasks（每日任务）
  ├── 1:N → share_records（分享记录，作为分享者）
  └── 1:N → admin_logs（管理员日志）

drift_bottles（漂流瓶）
  └── 1:N → salvage_records（被打捞记录）
  └── 1:N → bottle_likes（点赞记录）

4.2 各表字段

表名核心字段说明
users
用户表		 id, email(唯一), nickname(唯一), password_hash, referral_code(唯一), referrer_id, is_admin, created_at 密码哈希存储。referral_code 用于分享拉新链接。referrer_id 指向邀请人。
drift_bottles
漂流瓶表		 id, user_id, image_path, thumbnail_path, title, game_name, field_a, field_b, recommendation, category, is_approved, is_preset, created_at category 取 7 大分类 ID。is_preset 标记是否为系统预设瓶。is_approved 标记审核状态。
salvage_records
打捞记录表		 id, user_id, bottle_id, is_saved_to_wall, salvaged_at 联合唯一约束(user_id, bottle_id)：同一用户不能重复打捞同一个瓶子。
bottle_likes
点赞表		 id, user_id, bottle_id, created_at 联合唯一约束：同一用户只能点赞一次。再次点击取消点赞。
daily_tasks
每日任务表		 id, user_id, task_date, throw_claimed, share_claimed, referral_count 联合唯一约束(user_id, task_date)：每个用户每天一条记录。0:00 自动新建。
share_records
分享记录表		 id, sharer_id, shared_type, shared_target_id, new_user_id, created_at shared_type 取值：activity / bottle / wall。拉新成功时 new_user_id 不为空。
promo_codes
推广码表		 id, code(唯一), channel_name, max_uses, use_count, is_active, used_by, used_at max_uses=0 表示无限次。由光核各渠道分发。
admin_logs
管理员日志		 id, admin_id, action, target_type, target_id, note, created_at 记录审核通过/驳回/删除等操作，用于追溯。

五、核心业务逻辑

5.1 漂流瓶生命周期

用户投递 → 敏感词自动检测 → 审核（自动/人工）→ 进入漂流池
                                                ↓
                                          其他用户随机打捞
                                                ↓
                                    查看详情 / 点赞 / 收藏到安利墙
                                                ↓
                                          分享给好友（拉新）

5.2 打捞算法

  1. 排除该用户自己投的瓶子
  2. 排除该用户已经捞过的瓶子（数据库唯一约束保证）
  3. 优先抽取非预设瓶（真实用户投递的）
  4. 随机排序后取一个
  5. 如果非预设瓶被捞完了，回退到预设瓶

5.3 任务与奖励计算

获取途径奖励次数限制
基础额度1 次/天—
投递一个瓶子+1 次每日 1 次
分享活动/瓶子/安利墙+1 次每日首次自动领取
拉新成功（每人）+2 次每日最多 2 人

计算公式：

剩余次数 = min(基础1 + 投递奖励 + 分享奖励 + 拉新人数×2, 总上限) - 今日已用次数
每日 0:00 重置所有任务进度，未用次数不累积

5.4 分享拉新机制

环节说明
手机端分享调用 navigator.share()，弹出系统原生分享面板（微信/朋友圈/QQ等）
桌面端分享自动复制专属链接到剪贴板
链接格式/?via=tity（活动首页）、/s/bottle/3?via=tity（某个瓶子）、/s/wall/5?via=tity（某人安利墙）
拉新判定新用户注册时 URL 中带的 ?via= 参数对应分享者的 referral_code，自动记录邀请关系
验证方式信任制（H5 网页无法验证第三方 App 内的分享行为，此为行业通用做法）

七、安全与防刷

措施实现方式
CSRF 防护所有 POST 请求校验动态 token（session + header 双重验证）
密码安全Werkzeug generate_password_hash() 哈希存储，不明文保存
敏感词过滤投递时自动逐词匹配 sensitive_words.txt，命中则标记待审
内容审核管理员后台可查看待审列表，支持通过/驳回（附备注）/删除
投递冷却10 秒内禁止同一用户重复投递
打捞冷却2 秒内禁止同一用户重复打捞
重复打捞防护数据库联合唯一约束，同一用户不能捞同一个瓶子两次
拉新上限每人每日最多通过拉新获得 2×2=4 次额外打捞
图片安全仅允许 JPG/PNG/WebP 格式，最大 20MB，服务器端校验
